# torch.compile tuned-kernel findings on gfx1151 (N=8192)

## Architecture Summary

When using `torch.compile` for a FP8 mixed precision matmul on `gfx1151`, the framework does **not** generate a single fused kernel. Instead, it dispatches to three separate kernels:
1. **Pre-Cast (Triton):** Converts `B` from `fp8` (`f8e4m3fn` or `f8e5m2`) to `fp16` into intermediate global memory (`buf0`).
2. **Matmul (rocBLAS / Tensile):** A pure `fp16` x `fp16` matrix multiplication using highly optimized assembly (`extern_kernels.mm`).
3. **Epilogue (Triton):** A fused pointwise kernel applying scale and bias to the result.

This means that comparing our custom HIP FP8 mixed precision kernel (which fuses the cast inline) to `torch.compile` is actually a comparison against a **pure FP16 assembly kernel** (Tensile) that is fed pre-converted FP16 data.

## Tensile FP16 Matmul Analysis (The Baseline)

Through thread tracing (`--att`) of the Tensile FP16 kernel generated by `torch.compile`, we identified exactly why it hits ~32 TFLOPS (~53% theoretical peak), while our custom FP8 mixed precision kernel hits ~44.7 TFLOPS (~75% theoretical peak).

### Asynchronous Software Pipelining

The Tensile kernel successfully implements a perfectly interlocked asynchronous software pipeline.
- It issues `Global Load` instructions concurrently while the `WMMA` matrix cores are computing the previous chunk.
- The `Sync` (Wait for Global Memory) times are incredibly small because the data is prefetched well in advance.
- It pays **zero** `FP Convert` tax inside the loop, because the data was converted in a previous kernel launch.

### LDS Bloat and Dispatch Starvation

Despite having a perfect global memory pipeline, the Tensile kernel's internal math loop is heavily bottlenecked:
- **LDS Overhead:** Extracting cycle timings from the hardware trace shows that the Tensile kernel spends roughly **35% of its active compute time** just reading from LDS (`ds_read` operations) and calculating pointer arithmetic (`v_add`).
- **WMMA Starvation:** Because the dispatch pipe is clogged with thousands of narrow, unoptimized LDS reads and pointer instructions, the Matrix Cores (`WMMA`) are only active for **65%** of the math loop.
  - Maximum Possible TFLOPS due to LDS overhead: `59.4 TFLOPS * 65.0% = ~38.6 TFLOPS`.

### VGPR Occupancy Crash

Tensile kernels traditionally rely on massive register usage to hold state, prefetch buffers, and complex addressing logic.
- The selected Tensile config uses **256 VGPRs** per wave.
- A Workgroup of 4 waves requires `4 * 256 = 1024 VGPRs`.
- The RDNA3.5 SIMD limit is **1536 VGPRs**, meaning the hardware can only fit exactly **1 Workgroup per SIMD**.

Because occupancy is dropped to 1 Workgroup, if the kernel ever *does* stall, there are no other waves resident on the SIMD to hide the latency via hardware context switching.

## Why Our Custom HIP Kernel Wins

Our custom HIP kernel obliterates `torch.compile` end-to-end performance (~44.7 vs ~32 TFLOPS) because:
1. **Zero LDS Overhead:** We use wide `ds_read_b128` instructions and register caching to reduce LDS read time to **<1%** of the math loop, allowing our Matrix Cores to fire non-stop.
2. **High Occupancy:** By optimizing our VGPR footprint down to `192` registers, our `(2,2,2,4,4)` config fits `768 VGPRs` per Workgroup, allowing exactly **2 Workgroups per SIMD**. This provides "free" hardware latency hiding that perfectly masks our synchronous global memory loads.
3. **Fused Architecture:** We don't waste 1ms doing a full global memory pass to cast the `fp8` matrix before the matmul; we unpack it directly from LDS into VGPRs inline.
